# Cortex OS Task Policy

Cortex OS におけるタスク管理の方針を定義します。
Task Manager はこのポリシーに従ってタスクを選択・優先順位付けします。

---

## 基本原則

### 1. 認知負荷の最小化

- 人間の注意力は有限リソース
- 選択肢を減らすことが生産性を高める
- 「何をやらないか」を決めることが最重要

### 2. 1日3タスクルール

| 制約 | 理由 |
|------|------|
| 1日のタスクは最大3つ | 集中力の維持 |
| 深い仕事は1つまで | エネルギー配分 |
| 残りは軽いタスク | 達成感の確保 |

### 3. バッファの確保

- 予定の80%以上を埋めない
- 割り込み・予期せぬ作業に対応できる余白
- バッファなし = 破綻リスク

---

## タスク優先順位

### 優先度レベル

| レベル | 定義 | 例 |
|--------|------|-----|
| **P0** | 今日やらないと問題が起きる | デプロイ、障害対応、締切 |
| **P1** | 今週中に完了すべき | 機能実装、レビュー |
| **P2** | 今月中で良い | リファクタ、ドキュメント整備 |
| **P3** | いつかやる | 技術検証、改善アイデア |

### 選択アルゴリズム

```
1. P0 があれば最優先で選択
2. P1 から1つ選択（深い仕事枠）
3. P2/P3 から軽いタスクを補充
4. 合計3つになるまで
5. バッファ確認 → 超過なら削減
```

---

## タスクの分類

### 深い仕事（Deep Work）

- 90分以上の集中が必要
- 割り込みを禁止
- 1日1つまで

**例**:
- 新機能の設計・実装
- 複雑なバグ調査
- ドキュメント執筆
- アーキテクチャ検討

### 軽い仕事（Shallow Work）

- 30分以内で完了
- 割り込みOK
- 複数実行可能

**例**:
- コードレビュー
- Slack 返信
- 小さな修正
- 設定変更

---

## 1日の構造

### 推奨スケジュール

```
08:00 - 08:30  /brief 実行、今日のタスク確認
08:30 - 10:00  深い仕事（P1）
10:00 - 10:15  休憩
10:15 - 11:00  軽いタスク 1
11:00 - 12:00  軽いタスク 2 / バッファ
12:00 - 13:00  昼休み
13:00 - 17:00  残作業 / 予備 / 学習
17:00 - 17:30  /wrap-up 実行、振り返り
```

### 時間配分の目安

| 種類 | 時間 | 割合 |
|------|------|------|
| 深い仕事 | 90分 | 30% |
| 軽いタスク | 90分 | 30% |
| バッファ | 60分 | 20% |
| 休憩・移動 | 60分 | 20% |

```markdown

- [ ] P1タスク

---

## Backlog（いつか）

- [ ] P2/P3タスク

---

## Archive

### YYYY-MM-DD
- [x] 完了タスク
```

### 更新タイミング

| タイミング | 操作 |
|-----------|------|
| 朝（/brief後） | Today セクションを更新 |
| 夕（/wrap-up後） | 完了を Archive へ移動 |
| 週末（Weekly Summary後） | Backlog 整理 |

## ブロッカーの扱い

### ブロッカーとは

タスク完了を妨げる要因：
- 他者の承認待ち
- 情報不足
- 技術的障壁
- 環境準備

### 対処方針

1. **ブロッカーを明示** — タスク記述に `[blocked: 理由]` を追加
2. **解除アクションを定義** — 誰に何を依頼するか
3. **待機中は別タスクへ** — ブロック中のタスクに固執しない

---

## 週次レビュー

Weekly Summary（Recipe 11）と連動：

1. 完了率の確認
2. 未完了タスクの棚卸し
3. 来週のフォーカス決定
4. Backlog の整理（不要なものは削除）

---

## 禁止事項

- [ ] 4つ以上のタスクを Today に入れる
- [ ] 深い仕事を2つ以上計画する
- [ ] バッファなしのスケジュール
- [ ] ブロッカーを放置する
- [ ] 完了タスクを Archive に移動しない
